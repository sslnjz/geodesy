# CODING_RULES.md

## 1. Project target

This project is a C++17 geodesy library derived from the JavaScript geodesy library by Chris Veness. The implementation must be robust, readable, numerically stable, and compatible with the behaviour of the source library wherever practical.

C++ standard: **C++17 only**.

The codebase must be suitable for use as a library, not only as examples. Public APIs should be stable, predictable, and documented through names, comments, and tests.

## 3. Commenting rules

Core code must contain concise technical comments. Comments should help future maintainers verify geodesy formulae, unit conversions, constants, and numerical safeguards.

Required comments:

1. Public class purpose and module role.
2. Public function purpose when the name alone is not sufficient.
3. Parameters and return units where degrees/radians or metres/kilometres could be confused.
4. Non-obvious geodesy formulae, including haversine, Vincenty, Helmert transforms, n-vector operations, meridional arc calculations, UTM/MGRS grid logic, and OSGB conversions.
5. Boundary-condition branches, such as pole handling, anti-meridian wrapping, coincident points, near-antipodal points, and failed convergence.
6. Any tolerance value used for floating-point comparison or iteration termination.

Recommended style:

```cpp
// Convert public degree input to radians before applying the spherical formula.
const double phi1 = degToRad(lat1);

// Clamp the value before acos() to absorb rounding error near coincident points.
const double central = std::acos(clamp(cosCentral, -1.0, 1.0));
```

Forbidden in comments, documentation, commit messages, and test names:

- References to AI, Codex, ChatGPT, prompts, generated code, or automated authorship.
- Phrases such as `AI-generated`, `generated by assistant`, `as requested by prompt`, `LLM`, or `hallucinated`.
- Comments that describe obvious syntax instead of domain logic.
- Comments that excuse incomplete code.
- Placeholder comments that do not name the actual follow-up work.

Use technical English in source comments for consistency with the upstream geodesy library and common C++ conventions.

## 4. Formatting and style

1. Use 4 spaces for indentation. Do not use tabs.
2. Keep lines reasonably short; prefer under 120 columns unless a formula is clearer on one line.
3. Use Unix line endings.
4. Prefer `#pragma once` in headers after the required license block, unless existing include guards are already used consistently.
5. Include order:
   - Corresponding project header.
   - Other project headers.
   - Standard library headers.
   - Third-party headers.
6. Avoid `using namespace` in headers.
7. Avoid `using namespace std;` in all project code.
8. Keep implementation details in `.cpp` files or internal namespaces.
9. Keep public headers lightweight.
10. Do not introduce macros for constants or formulas. Use `constexpr`, `inline constexpr`, `enum class`, or functions.

Example header order:

```cpp
#pragma once

#include "geodesy/angle.h"
#include "geodesy/datum.h"

#include <cmath>
#include <optional>
#include <string>
```

## 5. Naming conventions

Use clear names that reflect geodesy terminology.

| Item           | Rule                                          | Example                            |
| -------------- | --------------------------------------------- | ---------------------------------- |
| Namespace      | lower-case                                    | `geodesy`                          |
| Class / struct | PascalCase                                    | `LatLon`, `Vector3d`, `OsGridRef`  |
| Function       | lowerCamelCase                                | `distanceTo`, `initialBearingTo`   |
| Variable       | lowerCamelCase                                | `semiMajorAxis`, `flattening`      |
| Constants      | lowerCamelCase or PascalCase for named datums | `meanEarthRadius`, `WGS84`         |
| Enum type      | PascalCase                                    | `DmsFormat`                        |
| Enum value     | PascalCase                                    | `DmsFormat::DegreesMinutesSeconds` |
| Test case      | descriptive behaviour name                    | `VincentyReturnsExpectedDistance`  |

Do not use cryptic one-letter names except where they directly match established mathematical notation and improve readability inside a short formula block. Greek-derived variable names from the JavaScript source may be translated to readable ASCII names when that improves maintainability, for example `phi`, `lambda`, `theta`, `delta`, `sigma`.

## 6. Types and data modelling

1. Use `double` for geodesy calculations unless a clear reason exists for another type.
2. Use `std::string` for owned text and `std::string_view` for read-only parameters where lifetime is safe.
3. Use `std::array<double, N>` for fixed-size numeric tuples only when a named struct would not be clearer.
4. Prefer named structs for ellipsoid, datum, transform, reference frame, and coordinate data.
5. Use `std::vector<T>` for variable-length collections such as polygon vertices.
6. Use `std::optional<T>` only when absence is an expected result.
7. Use `std::variant` sparingly and only for true sum types.
8. Avoid raw owning pointers. Use values, references, `std::unique_ptr`, or `std::shared_ptr` only when ownership semantics require them.
9. Mark non-mutating member functions `const`.
10. Mark simple value-returning functions `[[nodiscard]]` when ignoring the result is likely to be a bug.

## 7. Constants and units

1. Constants must be declared as `constexpr` or `inline constexpr` where possible.
2. Keep ellipsoid constants and transform parameters at reference precision.
3. Name units explicitly when ambiguity is possible: `distanceMetres`, `heightMetres`, `bearingDegrees`, `angleRadians`.
4. Public APIs use degrees for latitude, longitude, and bearings unless explicitly named otherwise.
5. Internal trigonometric calculations use radians.
6. Distances use metres unless an API explicitly documents another unit.
7. Heights use metres above the ellipsoid.
8. Epochs use decimal years when matching reference-frame transform definitions.
9. Do not silently convert kilometres, nautical miles, or feet unless the API explicitly supports those units.

Recommended helpers:

```cpp
inline constexpr double pi = 3.141592653589793238462643383279502884;
inline constexpr double degreesToRadians = pi / 180.0;
inline constexpr double radiansToDegrees = 180.0 / pi;
```

## 8. Angle wrapping and normalisation

Use centralised helpers for wrapping. Do not duplicate ad hoc modulus logic across modules.

Required behaviours:

1. Latitude should be validated or wrapped only according to the documented API.
2. Longitude should normally be normalised to `[-180, +180)`.
3. Bearing should normally be normalised to `[0, 360)`.
4. DMS formatting must preserve hemisphere semantics.
5. Anti-meridian cases must be tested explicitly.
6. Avoid `std::fmod` bugs around negative values by using a helper that documents the target interval.

Example helper names:

```cpp
[[nodiscard]] double wrap360(double degrees);
[[nodiscard]] double wrap180(double degrees);
[[nodiscard]] double wrap90(double degrees);
[[nodiscard]] double toRadians(double degrees);
[[nodiscard]] double toDegrees(double radians);
```

## 9. Floating-point comparisons

1. Do not compare calculated `double` values with `==` in tests or production code, except for exact sentinel values or input validation before calculation.
2. Use absolute tolerances for near-zero values.
3. Use relative tolerances for large distances or easting/northing values.
4. Define tolerance names by meaning, not by arbitrary number.
5. Keep tolerance values close to the algorithm or test that uses them.
6. Clamp inputs to inverse trig functions when rounding error could push them outside `[-1, 1]`.

Example:

```cpp
constexpr double angularTolerance = 1e-12;
constexpr double metreTolerance = 1e-6;
```

## 10. Error handling

Use explicit, predictable error handling.

1. Throw `std::invalid_argument` for invalid input values or malformed strings.
2. Throw `std::domain_error` for mathematically undefined operations.
3. Throw `std::runtime_error` for algorithm failure, such as Vincenty non-convergence, when no meaningful result exists.
4. Do not return default coordinates to mask errors.
5. Do not write errors to `stdout` or `stderr` from library code.
6. Error messages should identify the invalid field and expected range or format.
7. Parsing functions must reject ambiguous or malformed input rather than silently guessing.

## 11. Module-specific rules

### 11.1 DMS

- Preserve parsing flexibility for common DMS forms, including signed decimal degrees and hemisphere suffixes.
- Formatting must be deterministic and covered by tests.
- Document separator behaviour and rounding effects.
- Compass point output must match the expected precision and wrap behaviour.

### 11.2 Vector3d

- Keep vector operations numerically safe around zero-length vectors.
- Document whether a function returns a new vector or mutates the current object.
- Normalisation of a zero vector must be explicitly handled.
- Dot and cross products must not reorder operands casually because sign matters.

### 11.3 Spherical LatLon

- Preserve spherical model assumptions and default earth radius.
- Keep distance, bearing, final bearing, midpoint, intermediate point, destination, intersection, cross-track, along-track, rhumb-line, and polygon operations aligned with the reference implementation.
- Test anti-meridian paths, poles, coincident points, and small distances.

### 11.4 Ellipsoidal LatLon and Cartesian

- Keep latitude, longitude, and height semantics clear.
- Cartesian conversion must use the correct ellipsoid parameters.
- Datum conversion must not mutate the source coordinate unless the API explicitly says so.
- Preserve WGS84 as the default datum where applicable.

### 11.5 Datum and Helmert transforms

- Store transform parameters with names and units: translations in metres, rotations in arc-seconds, scale in parts per million.
- Convert rotations and scale exactly once at the calculation boundary.
- Clearly document transform direction and inverse transform construction.
- Preserve datum constants and ellipsoid definitions.

### 11.6 Reference frames

- Preserve frame names, epochs, and transform parameters.
- Epoch-aware transforms must document whether parameters are static or rate-based.
- Avoid silent fallback when a frame transform is unsupported.

### 11.7 Vincenty

- Implement both direct and inverse algorithms with explicit iteration limits.
- Document convergence criteria and failure handling.
- Preserve near-antipodal and coincident-point behaviour through tests.
- Distinguish initial bearing and final bearing carefully.

### 11.8 N-vector

- Keep n-vector normalisation explicit.
- Document path and great-circle operations because sign conventions are easy to invert.
- Test pole and anti-meridian cases.

### 11.9 UTM

- Implement Norway and Svalbard special zones exactly.
- Validate zone, hemisphere, easting, and northing.
- Preserve false easting/northing logic.
- Use named constants for scale factor, false easting, false northing, and WGS84 parameters.

### 11.10 MGRS

- Validate grid zone designator, band, 100 km grid square, easting, northing, and precision.
- Preserve precision-dependent formatting.
- Reject invalid letters and impossible zone/band combinations.
- Document the relationship between MGRS and UTM conversion.

### 11.11 OS Grid Reference

- Preserve OSGB36 and WGS84 conversion behaviour.
- Document meridional arc iteration and Airy 1830 ellipsoid constants.
- Validate easting/northing and grid-reference strings.
- Test examples at multiple formatting precisions.

## 12. Public API design

1. Constructors must leave objects in a valid state.
2. Avoid implicit unit conversions in constructors.
3. Use named factories for parsing and conversions, such as `parse`, `fromCartesian`, `fromUtm`, where appropriate.
4. Use member functions for operations naturally performed from one coordinate to another, such as `distanceTo`.
5. Use free functions only for utilities that do not belong to a single type.
6. Prefer returning new values over mutating existing objects.
7. Document any function that mutates state.
8. Keep overloads unambiguous.
9. Avoid APIs that require users to pass magic booleans; prefer enums.
10. Public formatting should be deterministic and tested.

## 13. Headers and dependencies

1. Public headers must compile independently.
2. Keep include dependencies minimal.
3. Forward declare where it reduces compile cost and does not obscure ownership.
4. Do not include test-only headers in production code.
5. Do not add large third-party dependencies for parsing, formatting, or math that can be handled safely with the C++17 standard library.
6. Use CMake targets to express dependencies instead of global include paths when possible.

## 14. Testing rules

1. Every public function must have at least one direct test or be covered through a higher-level integration test.
2. Every bug fix must include a regression test.
3. Use fixed input/output examples from the reference implementation wherever possible.
4. Use meaningful tolerances, not overly broad tolerances that hide defects.
5. Test invalid inputs and parsing failures.
6. Test numerical edge cases: poles, anti-meridian, coincident points, near-antipodal points, zero distances, high precision DMS, invalid zones, invalid grid bands, and failed convergence.
7. Test formatting outputs exactly where deterministic strings are expected.
8. Tests must not depend on network access.
9. Tests must not depend on the local machine timezone or locale.
10. Keep tests deterministic and independent.

## 15. Build rules

1. The project must build with a C++17 compiler.
2. The default build should not require optional tools.
3. Warnings should be treated as useful defects; fix warnings in touched code.
4. Do not suppress warnings globally unless there is a documented reason.
5. Keep CMake target names clear and stable.
6. Separate library, tests, and examples into distinct targets when practical.
7. Do not hard-code machine-specific paths.
8. Do not require network access during build or test.

Recommended CMake baseline:

```cmake
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)
```

## 16. Refactoring rules

1. Refactor only with a clear behavioural or maintainability purpose.
2. Keep public API changes deliberate and documented.
3. Do not mix broad refactoring with unrelated bug fixes.
4. Preserve test coverage before moving code.
5. Prefer small internal helpers over duplicated calculations.
6. Remove dead code only after confirming it is unused by tests, examples, and public headers.
7. Avoid changing formatting output unless the previous output is demonstrably wrong.
8. Keep compatibility with the reference behaviour more important than personal style preference.

## 17. Documentation rules

1. Document public classes and functions in headers when the behaviour is not obvious.
2. Include units in public documentation.
3. Include valid ranges for latitude, longitude, bearing, zone, easting, northing, and precision fields.
4. Include exception behaviour for parsing and invalid input.
5. Keep examples short and executable when possible.
6. Do not mention tools, prompts, or generated authorship.

## 18. Security and robustness

1. Parsing code must handle malformed input without crashes.
2. Avoid unchecked indexing into strings, vectors, and arrays.
3. Validate numeric conversions and reject `NaN` or infinite values for public coordinate inputs unless explicitly supported.
4. Avoid undefined behaviour in arithmetic and casts.
5. Do not use C-style string mutation or unsafe buffer functions.
6. Do not add runtime network calls.
7. Do not add filesystem access to core library functions unless explicitly required.

## 19. Completion checklist

Before marking a coding task complete:

- Required header is present in every touched source file.
- Core algorithms and edge-case branches have natural technical comments.
- No comments or documentation contain AI/tooling traces.
- Public API remains compatible or the change is clearly justified.
- Build succeeds.
- Tests pass.
- New or changed behaviour is covered by tests.
- Numerical tolerances are meaningful and not overly broad.
- Formatting output changes are intentional and tested.
- No unrelated features or dependencies were added.
